Koml

On-device LLM inference for Kotlin Multiplatform. A thin, idiomatic wrapper over llama.cpp. Coroutines and Flow on the surface, GGUF underneath. Same code runs on Android, iOS, JVM desktop, and macOS native.

Get started → Browse the API ↗

Why Koml

llama.cpp already solves the hard part — loading quantised transformers from GGUF, doing the math fast, managing the KV cache. What's been missing is a Kotlin-shaped seam over it: one that feels native to the idioms Kotlin developers already use, doesn't leak C-level abstractions, and works the same way on every target without per-platform forking.

That's what Koml is.

Flow<TokenEvent> streaming

Cancel the collecting coroutine and generation stops at the next sampling iteration. No threads to manage.

One API, four targets

Identical entry point on Android, iOS, JVM desktop, and macOS native.

Curated model registry

Five vetted ungated GGUF models bundled in. Hugging Face Hub search for everything else.

Resumable downloads

SHA-256 verify-as-you-write, atomic rename on success, HTTP Range resume across process restarts.

Per-family chat templates

ChatML, Llama 3, Phi-3, Gemma — each rendered from the spec on its HuggingFace card.

Boring and correct

Sealed errors, default-impl interface methods for backward compat, KDoc on every public type.

Install

One line in your build.gradle.kts:

dependencies {
    implementation("io.github.12345debdut:engine-llama:0.0.5")
}

The engine-llama artifact transitively pulls in :core, :storage, :download, and :registry — one dependency line is the whole library.

Kotlin Multiplatform consumer

kotlin {
    sourceSets {
        commonMain.dependencies {
            implementation("io.github.12345debdut:engine-llama:0.0.5")
        }
    }
}

KMP auto-resolves to the per-target artifact (engine-llama-jvm, engine-llama-iosarm64, engine-llama-macosarm64, etc.) at link time.

iOS (Swift) via XCFramework

v0.0.5 doesn't ship CocoaPods or SPM yet — use the XCFramework directly. See the SwiftUI sample for the Xcode setup walkthrough (XcodeGen-based).

Quickstart

End-to-end on any target — Android, JVM, macOS Native — the code is identical:

import dev.koml.engine.LlmKit
import dev.koml.core.config.RuntimeConfig
import dev.koml.core.download.DownloadState
import dev.koml.core.session.*
import kotlinx.coroutines.flow.*

suspend fun runIt() {
    // 1. Wire up the coordinator (storage, registry, downloader, license gate).
    val coordinator = LlmKit.initialize()

    // 2. Pick a model from the bundled curated registry.
    val model = coordinator.registry.curated()
        .first { it.id == "smollm2-135m-instruct-q8" }

    // 3. Download (resumable, SHA-256 verified, ~145 MB for SmolLM2-135M).
    val handle = coordinator.downloader.download(model)
        .onEach { state ->
            if (state is DownloadState.Progress) {
                println("${state.bytesDownloaded} / ${state.totalBytes}")
            }
        }
        .filterIsInstance<DownloadState.Completed>()
        .first()
        .handle

    // 4. Load the model.
    val session = coordinator.loadModel(handle, RuntimeConfig(contextSize = 2048))

    // 5. Stream tokens. session.chat() picks the right template from the
    //    model (here: ChatML for SmolLM2). session.generate() takes a raw
    //    prompt if you don't want template formatting.
    session.chat(
        listOf(
            ChatMessage(ChatRole.System, "You are concise. One-sentence answers."),
            ChatMessage(ChatRole.User, "Why is the sky blue?"),
        ),
        GenParams(maxTokens = 128),
    ).collect { event ->
        when (event) {
            is TokenEvent.Token -> print(event.text)
            is TokenEvent.Done  -> println(
                "\n[${event.reason}, ${event.stats.generatedTokens} tokens, " +
                "%.1f tok/s]".format(event.stats.tokensPerSecond)
            )
            is TokenEvent.Error -> println("\nError: ${event.cause.message}")
        }
    }

    // 6. Release the native model + context.
    session.unload()
}

Cancellation is automatic via Flow — drop the collecting coroutine and generation stops at the next iteration.

Troubleshooting

Build fails with "ggml.h not found" or "unknown target CPU 'apple-m3'"

On a recent Apple Silicon host (M3, M4) llama.cpp's auto-CPU-detection asks AppleClang to target a CPU it doesn't yet recognise. All three Koml build scripts pass -DGGML_NATIVE=OFF to disable auto-detection and use a generic arm64 target instead.

Metal shader fails at runtime with "block_q4_0 not declared"

This is the b5460 ggml-common embed bug. All Apple targets in v0.0.x ship with Metal disabled (CPU + Accelerate only). Returns in v0.1.0 with a llama.cpp version bump.

HF download fails with "SHA-256 mismatch"

HuggingFace re-quantised the file you're downloading. Run ./scripts/refresh-manifest-shas.sh from a clone of this repo (or update the SHA-256 in your own consumer code).

Generation throws "Decode failed (context window exceeded)"

Your prompt + history is longer than the model's contextWindow. Either truncate older messages or load with a larger RuntimeConfig.contextSize (capped at ModelInfo.contextWindow).

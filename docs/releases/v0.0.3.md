# Koml v0.0.3 ‚Äî chat, search, native

*Released: 2026-06-05*

> `session.chat()` works now. So does HuggingFace search. So does Kotlin/Native on macOS. So do tests. The library is feature-complete for v1 except for the publishing infrastructure.

v0.0.3 closes the four biggest gaps that were keeping Koml a "demo library":

1. **Chat is a first-class call.** `session.chat(messages)` knows the right format for ChatML, Llama 3, Phi-3, and Gemma. Drop in a `List<ChatMessage>`, get back a `Flow<TokenEvent>`.
2. **You can find models you don't have.** `coordinator.registry.searchHuggingFace("smollm2")` calls the real Hub API.
3. **Kotlin/Native works.** macOS arm64 + x64 targets are live ‚Äî use Koml from a Native CLI tool with the same `LlmKit.initialize()` entry point.
4. **There are tests.** Ktor `MockEngine` + Okio `FakeFileSystem` exercise the downloader's resume/SHA-256/license-gate logic in milliseconds, no real network or model load.

## What's new

### `session.chat(messages)` for real

```kotlin
val coordinator = LlmKit.initialize()
val handle = coordinator.downloader.localModels()
    .first { it.info.id == "phi-3-mini-4k-instruct-q4" }
val session = coordinator.loadModel(handle)

session.chat(
    listOf(
        ChatMessage(ChatRole.System, "You are concise."),
        ChatMessage(ChatRole.User, "Why is the sky blue?"),
    ),
).collect { event ->
    if (event is TokenEvent.Token) print(event.text)
}
```

Behind the scenes a sealed `ChatTemplate` picks the right format from `ModelInfo.promptTemplate`. Stop sequences come from the template too (`<|im_end|>`, `<|eot_id|>`, `<|end|>`, `<end_of_turn>`) and are merged with any in `GenParams.stopSequences`, so turns terminate cleanly without you having to memorise each family's special tokens.

| Family | Models that use it (in our curated list) |
|---|---|
| ChatML | SmolLM2-135M, SmolLM2-1.7B, TinyLlama-1.1B-Chat, Qwen2.5-1.5B |
| Phi-3 | Phi-3-mini-4k-instruct |
| Llama 3 | (no Llama 3 in the curated list yet ‚Äî template ready) |
| Gemma | (no Gemma in the curated list yet ‚Äî template ready) |
| None | base / non-instruct models |

### HuggingFace Hub search

```kotlin
val coordinator = LlmKit.initialize()
val results = coordinator.registry.searchHuggingFace("phi-3-mini")
results.take(5).forEach { println("${it.displayName} ‚Äî ${it.license.spdxId}") }
```

Returns up to 20 results sorted by downloads. **Metadata-only** ‚Äî `downloadUrl` and `sha256` are blank since we deliberately avoid an N+1 per-repo file-list fetch (a search of 20 results would otherwise be 21 API hits). Callers who want to *download* a search hit need to supplement those fields themselves; this is documented on `ModelRegistry.searchHuggingFace`.

Errors degrade to an empty list ‚Äî rate limits, server errors, parse failures all silently return nothing instead of throwing. The curated list always works.

### Kotlin/Native on macOS

```kotlin
// In a Kotlin/Native CLI's commonMain
implementation("io.github.12345debdut:engine-llama:0.0.3")

fun main() = runBlocking {
    val coordinator = LlmKit.initialize()
    val handle = coordinator.downloader.localModels().first()
    val session = coordinator.loadModel(handle, RuntimeConfig(contextSize = 2048))
    session.generate("Explain coroutines in one paragraph.")
        .collect { event ->
            if (event is TokenEvent.Token) print(event.text)
        }
}
```

Both `macosArm64` and `macosX64` targets are exposed on `:engine-llama` (and every dependency module). One-time setup: `./scripts/build-llama-macos-native.sh` produces the per-arch combined static archive, then cinterop and link work like the iOS path.

**Metal works on macOS native** ‚Äî the b5460 ggml-common embed bug that disables Metal on iOS doesn't bite macOS. Expect significant speedup over the CPU-only iOS path.

### Tests

```bash
./gradlew :download:jvmTest        # 6 tests, ~250 ms
./gradlew :engine-llama:jvmTest    # 7 tests, ~100 ms
```

Coverage: resumable download algorithm, SHA-256 mismatch handling, license gate, already-downloaded short-circuit, chat template rendering for all five families. Backed by Ktor `MockEngine` + Okio `FakeFileSystem` ‚Äî no network, no disk, no real GGUF, no llama.cpp invocation.

### KDoc on every public type

Every public type in `:core` now has a doc comment that explains what it is, what it does, and how it interacts with the rest of the API. Hover-tooltips in your IDE work. Dokka-generated HTML will look professional. Worth opening `LlmCoordinator.kt`, `LlmSession.kt`, or `KomlException.kt` to see the level of detail.

### Better error messages

Generation errors now carry the model id + prompt token count + context window so the message itself tells you what to fix:

```
Decode failed for prompt (3081 tokens) on model 'smollm2-135m-instruct-q8'
‚Äî context window (2048) may be exceeded.
```

## What hasn't changed

- The `:core` public API surface ‚Äî fully backward-compatible with v0.0.1 and v0.0.2.
- The `:samples-android` / `:samples-desktop` / `samples-ios/` flows. They'll automatically pick up chat templates when you switch a `session.generate(prompt)` call to `session.chat(messages)`.
- The b5460 iOS Metal embed bug ‚Äî still bypassed via CPU + Accelerate on iOS. macOS native is the workaround host for Metal-accelerated llama.cpp.

## Getting started

Adds one new one-time step on top of v0.0.2's setup:

```bash
# Kotlin/Native macOS static libs (~10 min)
./scripts/build-llama-macos-native.sh
```

Then everything from v0.0.2 still works.

## Known limitations (carried over)

- **No Metal on iOS** ‚Äî b5460 embed bug. macOS targets do get Metal.
- **Curated manifest SHA-256s** are still placeholders shipped in source. Run `scripts/refresh-manifest-shas.sh`.
- **Linux/Windows JVM** still build-from-source. CI in v0.0.4.
- **Linux Kotlin/Native** ‚Äî deferred to v0.0.4 (needs Docker cross-build or Linux CI runner).

New in v0.0.3:

- **HF search results aren't directly downloadable** ‚Äî see `ModelRegistry.searchHuggingFace` KDoc. Will get an opt-in `withFileDetails` flag in a later release.

## Breaking changes

None. `DefaultModelRegistry` gained an optional constructor parameter but the no-arg form still works exactly as before. All `:core` interfaces are source- and binary-compatible.

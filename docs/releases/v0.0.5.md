# Koml v0.0.5 — pre-1.0 polish

*Released: 2026-06-05*

> The high-priority items between v0.0.4 and 1.0 land here. No breaking changes, no new platforms — just the items the original v1 brief expected to see settled before claiming the API frozen.

## What's new

### HuggingFace search results can now be downloaded

```kotlin
val coordinator = LlmKit.initialize()

// Cheap: metadata only (~1 API hit total)
val previews = coordinator.registry.searchHuggingFace("smollm2")

// Pricier but each result is directly downloadable (~N+1 API hits)
val ready = coordinator.registry.searchHuggingFaceWithDetails("smollm2")
val first = ready.first()
coordinator.downloader.download(first).collect { /* … */ }
```

The new method on `ModelRegistry` fans out one `GET /api/models/<id>` per result and picks the first `.gguf` from the siblings list (preferring `Q4_K_M` variants for size/quality balance), populating `downloadUrl`, `sha256` (when HF exposes it), and `sizeBytes`. Default impl preserves backward compat — old `ModelRegistry` consumers keep working.

### Dokka HTML site

```bash
./gradlew dokkaHtmlMultiModule
open build/dokka/htmlMultiModule/index.html
```

Aggregates KDoc from `:core`, `:storage`, `:download`, `:registry` into a single browsable site. `:engine-llama` is skipped (its native source sets need user-supplied static libs to resolve cinterop) — its public surface is small and well-documented in source.

A new `.github/workflows/docs.yml` workflow publishes the site to GitHub Pages on every push to `main` that touches Kotlin source.

### Stop-sequence detection extracted + edge cases fixed

The inline stop-sequence detector had two latent bugs:

- **Empty stop sequences** (`""`) silently matched every token, so passing user input through could nuke generation.
- **Stop sequences longer than `MAX_STOP_BUFFER`** (256) could never match.

`StopSequenceDetector` is now a small class with 9 tests covering split-across-tokens, longer-than-buffer, first-match-wins-by-declaration-order, trimming-while-keeping-tail-intact, and the two bugs above.

### Stop-sequence rework is also a tiny perf win

The old loop allocated a fresh `firstOrNull` lambda per token *and* did a list scan against `params.stopSequences`. The new loop captures the filtered (non-empty) list once at construction and the per-token cost is a single `endsWith` per active sequence. Difference is sub-millisecond per token but it adds up.

### Real-GGUF JVM integration test (opt-in)

`engine-llama/src/jvmTest/.../JvmEndToEndTest.kt` covers the full pipeline:
1. Resolves SmolLM2-135M from the curated registry.
2. Downloads it (real HF, real SHA-256 verification).
3. Loads it.
4. Generates 5 tokens.
5. Asserts at least one non-empty token + no errors.

Gated on `KOML_INTEGRATION_TESTS=1` so CI doesn't pay the ~2 min + 145 MB cost on every PR. Maintainers run before each release:

```bash
KOML_INTEGRATION_TESTS=1 ./gradlew :engine-llama:jvmTest \
    --tests "dev.koml.engine.integration.*"
```

This is the test that catches "the mocks all pass but llama.cpp itself doesn't actually decode" — the regression class that's been theoretical until now.

### CI-driven publishing on tag push

`.github/workflows/publish.yml`:
- Triggers on push of any `v*` tag, or manual `workflow_dispatch` with a tag input.
- Runs on `macos-latest` (need macOS for the iOS + JVM JNI builds).
- Builds all three sets of llama.cpp static libs, runs the test gate, signs with the GPG key from secrets, stages to Central Portal via `publishAggregationToCentralPortal`.
- Leaves the bundle in **VALIDATED** state for human review — no auto-publish, on purpose.

Required secrets: `MAVEN_CENTRAL_USERNAME`, `MAVEN_CENTRAL_PASSWORD`, `SIGNING_IN_MEMORY_KEY`, `SIGNING_IN_MEMORY_KEY_PASSWORD`. The first manual publish (per `docs/PUBLISHING.md`) still needs to happen before this workflow is useful.

### Metal-on-iOS + Windows JVM: decisions documented

Both items get **decisions** in [`docs/known-issues.md`](../known-issues.md), not fixes:

- **Metal on iOS** is officially deferred until the next llama.cpp version bump (which is a minor-version event per VERSIONING.md). v0.0.x ships CPU-only on iOS — that's the supported configuration.
- **Windows JVM** is officially deferred past 1.0. Linux is in the same bucket. Both need additional CI infrastructure; neither is on the v1 critical path. v1.x is the natural moment when there's demand.

This isn't a feature; it's clarity. Future contributors can stop wondering whether these are open questions.

## What hasn't changed

- Every public API in `:core`. v0.0.4 code keeps working.
- The five curated models, the chat templates, the cinterop bindings.
- Platform coverage: still Android, iOS, macOS JVM, macOS Kotlin/Native.

## Maven Central status

Same as v0.0.4: pipeline is wired and tested locally; first actual publish still pending. After it lands once via `./gradlew publishAggregationToCentralPortal`, the new `.github/workflows/publish.yml` makes subsequent releases a `git tag v0.0.6 && git push origin v0.0.6` away.

## Breaking changes

None. `ModelRegistry` gained a new method with a default implementation that delegates to the existing `searchHuggingFace` — backward-compatible per [VERSIONING.md](../VERSIONING.md).

# Koml v0.0.1 — first developer preview

*Released: 2026-06-04*

> On-device LLM inference for Kotlin Multiplatform — a thin, idiomatic wrapper over llama.cpp with Flow-based streaming.

This is the first public preview of Koml. It establishes the public API and ships working engines on **Android and iOS**, with JVM desktop and native targets landing in upcoming releases.

The goal of v0.0.1 isn't feature completeness — it's to lock the public API surface so we can iterate underneath without breaking your code, and to prove the cross-platform streaming story really does feel like one library and not three.

## What's in this release

### A single Kotlin Multiplatform API for on-device inference

```kotlin
val coordinator = LlmKit.initialize()
val session = coordinator.loadModel(
    ModelHandle(info = ..., localPath = "/path/to/model.gguf"),
    RuntimeConfig(contextSize = 2048),
)

session.generate(
    prompt = "The capital of France is",
    params = GenParams(maxTokens = 64, temperature = 0.7f),
).collect { event ->
    when (event) {
        is TokenEvent.Token -> print(event.text)
        is TokenEvent.Done  -> println("\n[${event.reason}, ${event.stats.tokensPerSecond} tok/s]")
        is TokenEvent.Error -> println("\nError: ${event.cause.message}")
    }
}
```

The same code paths run on Android (JNI to llama.cpp) and iOS (Kotlin/Native cinterop to llama.cpp). One `:engine-llama` artifact; one set of public types in `:core`.

### Flow-based streaming with proper cancellation

Generation is a `Flow<TokenEvent>` that emits one `Token` per sampled token, terminating with a `Done` event carrying `FinishReason`, total tokens, and tokens/sec. Cancel the collecting coroutine and generation stops at the next sampling iteration — no zombie threads, no half-decoded KV cache.

### Per-session single-threaded dispatcher

Each `LlmSession` gets its own `Dispatchers.Default.limitedParallelism(1)` view. Two sessions running in parallel never touch the same `llama_context` from two threads. Designed in from day one because retrofitting threading is painful.

### iOS that feels like iOS

SwiftUI sample app with `for await event in flow` directly over the Kotlin `Flow`. Exhaustive sealed-class switches via SKIE's `onEnum(of:)`. Kotlin `suspend fun` exposed as Swift `async throws`. The bridging is opaque so Swift developers don't have to think about it.

```swift
let flow = session.generate(prompt: "...", params: GenParams(...))
for await event in flow {
    switch onEnum(of: event) {
    case .token(let token): output += token.text
    case .done(let done):   print("\n[\(done.reason), \(done.stats.tokensPerSecond) tok/s]")
    case .error(let err):   print("\nError: \(err.cause.message ?? "")")
    }
}
```

### Sealed error hierarchy

All failures land as `KomlException` subclasses: `ModelLoadException`, `InferenceException`, `DownloadException`, `StorageException`, `EngineNotAvailableException`, `LicenseNotAcceptedException`. No string-matching on exception messages, no generic `RuntimeException` leaks at the API boundary.

## Platform support

| Target | Status | Acceleration |
|---|---|---|
| Android (arm64-v8a) | ✅ Ready | CPU |
| iOS arm64 (device + simulator) | ✅ Ready | CPU + Accelerate BLAS |
| iOS x86_64 simulator | ✅ Ready | CPU |
| JVM desktop (macOS, Linux) | ⏳ next release | — |
| Native (macOS, Linux) | ⏳ planned | — |

llama.cpp pinned to **tag b5460**, vendored as a git submodule. GGUF models load directly from disk — no conversion step.

## Getting started

```bash
git clone --recurse-submodules https://github.com/<you>/koml.git
cd koml

# One-time iOS prerequisite (~10 min)
./scripts/build-llama-ios.sh

# Android sample
./gradlew :samples-android:assembleDebug
adb push your-model.gguf /data/local/tmp/model.gguf
adb install -r samples-android/build/outputs/apk/debug/samples-android-debug.apk

# iOS sample (SwiftUI)
brew install xcodegen
./gradlew :engine-llama:assembleKomlEngineDebugXCFramework
cd samples-ios
# Drop a .gguf at KomlSample/Resources/model.gguf
xcodegen && open KomlSample.xcodeproj
```

Verified end-to-end with **SmolLM2-135M Q8_0** (~145 MB) — small enough to iterate on, real enough to demonstrate streaming. TinyLlama-1.1B, Llama 3.2 1B, Phi-3 Mini, and Gemma 2 2B are tested-known-good too.

## Known limitations

| Limitation | Workaround / plan |
|---|---|
| **No Metal on iOS** | Disabled due to a b5460-specific ggml-common embed bug in Metal shader sources. iOS runs on CPU with Apple's Accelerate BLAS — plenty fast for sub-3B models. Tracked for the llama.cpp version bump. |
| **No download / no registry** | `coordinator.registry.curated()` returns `emptyList()`. `coordinator.downloader.download(...)` emits `DownloadState.Failed`. Place models on disk yourself for now; full download flow lands in the next release. |
| **No chat templates** | `session.chat(...)` throws `NotImplementedError`. Use `session.generate(...)` with prompts formatted for your model. chatml / llama3 / phi3 / gemma templates land in a later release. |
| **AGP 9 bypass** | Koml sets `android.builtInKotlin=false` + `android.newDsl=false` to keep `com.android.library` + `kotlin-multiplatform` together until the new `com.android.kotlin.multiplatform.library` DSL matures for NDK use cases. Transparent to consumers. |
| **Android: arm64-v8a only** | Other ABIs land alongside the JVM target. |

Full audit in [`docs/known-issues.md`](../known-issues.md).

## Breaking changes

None — this is the first release. From v0.0.1 onward, the public API in `:core` is **additive only**. Sealed classes will gain new subclasses (a non-binary-compatible change you should review on each minor release while we're in 0.x), but no existing types or signatures will be renamed without a deprecation cycle.

## What's next (distribution layer)
